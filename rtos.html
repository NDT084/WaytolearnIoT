RTOS : Systèmes d'exploitation Temps Réel

Maîtrisez les systèmes d'exploitation temps réel pour garantir le déterminisme et la prévisibilité dans vos applications IoT.

Objectifs pédagogiques : Comprendre pourquoi un RTOS est nécessaire en IoT, maîtriser tâches, priorités et synchronisation, implémenter FreeRTOS sur ESP32/STM32, et garantir le déterminisme temporel.

1. Pourquoi un RTOS en IoT ?

Un système IoT doit souvent lire des capteurs, communiquer, afficher des données et réagir aux interruptions en parallèle, ce qui dépasse vite une simple boucle while(1).

Problème sans RTOS (Super Loop)

void loop() {
    readSensor();      // Prend 50ms
    delay(1000);       // Attend 1 seconde
    sendData();        // Prend 200ms
    delay(1000);       // Attend encore 1 seconde
    // ⚠️ Problème : Si une interruption arrive 
    // pendant delay(), elle est ignorée !
}

Limitation : Réponse lente aux événements critiques, structure difficile à maintenir et à faire évoluer.

Solution avec RTOS (Tâches concurrentes)

void taskSensor(void *pvParameters) {
    while(1) {
        readSensor();
        vTaskDelay(1000 / portTICK_PERIOD_MS);
    }
}

void taskCommunication(void *pvParameters) {
    while(1) {
        sendData();
        vTaskDelay(1000 / portTICK_PERIOD_MS);
    }
}
// Les deux tâches s'exécutent en parallèle,
// l'ordonnanceur partage le CPU.

Avantage : Meilleure réactivité, code plus modulaire et comportement plus prévisible.

2. Concepts Fondamentaux du RTOS

Tâches (Tasks)

Une tâche est une fonction planifiée par le RTOS, avec sa propre pile et éventuellement sa priorité.

Exemple FreeRTOS :

xTaskCreate(
    taskFunction,    // Fonction
    "TaskName",      // Nom
    1024,            // Stack size
    NULL,            // Paramètres
    2,               // Priorité
    NULL             // Handle
);

Priorités

L’ordonnanceur exécute toujours la tâche prête la plus prioritaire, ce qui permet de respecter des contraintes temps réel.

Exemple : Une tâche capteur critique (prio 3) peut préempter une tâche de communication (prio 2) ou d’affichage (prio 1).

Context Switching

Le changement de contexte sauvegarde l’état d’une tâche et restaure celui d’une autre en quelques microsecondes sur MCU moderne.

Impact : Le coût reste faible par rapport aux périodes d’échantillonnage habituelles en IoT.

3. Synchronisation : Sémaphores, Mutex et Queues

Les mécanismes de synchronisation évitent les races et les corruptions de données lorsque plusieurs tâches partagent une même ressource.

Mutex (Mutual Exclusion)

Un mutex garantit qu’une seule tâche à la fois entre dans une section critique (SPI, variable partagée, etc.).

Exemple pratique :

SemaphoreHandle_t mutex = xSemaphoreCreateMutex();

void task1() {
    xSemaphoreTake(mutex, portMAX_DELAY);
    // Zone critique : accès exclusif
    sharedVariable++;
    xSemaphoreGive(mutex);
}

Usage : Protection d’accès à une ressource partagée entre plusieurs tâches.

Queues (Files d'attente)

Les queues permettent à des tâches de s’échanger des messages ou structures de manière sûre.

Exemple pratique :

QueueHandle_t queue = xQueueCreate(10, sizeof(int));

// Tâche productrice
void taskSensor() {
    int data = readSensor();
    xQueueSend(queue, &data, 0);
}

// Tâche consommatrice
void taskDisplay() {
    int data;
    xQueueReceive(queue, &data, portMAX_DELAY);
    display(data);
}

Usage : Chaîner capteurs, communication réseau et affichage sans variables globales.

4. FreeRTOS vs Zephyr : Comparaison pour l'IoT

FreeRTOS

RTOS très répandu, minimaliste, soutenu par AWS, avec un noyau léger adapté aux microcontrôleurs contraints. [web:65][web:68]

  • ✓ Avantages : Faible empreinte, API simple, énorme base d’exemples (ESP32, STM32, etc.). [web:68]
  • ✓ Idéal pour : Projets éducatifs, prototypes, produits simples à moyenne complexité. [web:65]

Ressource : www.freertos.org

Zephyr RTOS

RTOS modulaire soutenu par la Linux Foundation, avec un écosystème riche, réseau intégré et forte orientation sécurité. [web:65][web:68]

  • ✓ Avantages : Stack réseau avancée, drivers nombreux, intégration Bluetooth/WiFi et sécurité renforcée. [web:68]
  • ✓ Idéal pour : Projets industriels, produits certifiables et architectures complexes. [web:65]

Ressource : www.zephyrproject.org

5. Exemple Pratique : Station Météo IoT avec FreeRTOS

Exemple d’architecture FreeRTOS sur ESP32 : tâches capteurs, WiFi/MQTT et affichage OLED fonctionnent de manière découplée.

Architecture du système

  • Tâche 1 (Prio 3) : Lit les capteurs toutes les 5 s et pousse dans une queue.
  • Tâche 2 (Prio 2) : Envoie les données via WiFi/MQTT.
  • Tâche 3 (Prio 1) : Met à jour l’OLED avec les dernières valeurs.
  • ISR : Bouton d’urgence qui réveille une tâche de traitement.

Code FreeRTOS (simplifié)

// Queue pour les données
QueueHandle_t sensorQueue;

void taskSensor(void *pvParameters) {
    while(1) {
        float temp = readTemperature();
        float humidity = readHumidity();
        
        SensorData data = {temp, humidity};
        xQueueSend(sensorQueue, &data, 0);
        
        vTaskDelay(5000 / portTICK_PERIOD_MS);
    }
}

void taskWiFi(void *pvParameters) {
    SensorData data;
    while(1) {
        if(xQueueReceive(sensorQueue, &data, portMAX_DELAY)) {
            mqttPublish("temp", data.temp);
            mqttPublish("humidity", data.humidity);
        }
    }
}

void setup() {
    sensorQueue = xQueueCreate(10, sizeof(SensorData));
    xTaskCreate(taskSensor, "Sensor", 2048, NULL, 3, NULL);
    xTaskCreate(taskWiFi, "WiFi", 4096, NULL, 2, NULL);
    vTaskStartScheduler();
}

Analyse des performances

  • Déterminisme : La période de 5 s est contrôlée via le tick RTOS, pas des delays bloquants.
  • Réactivité : La communication lente ne bloque ni lecture capteur ni affichage.
  • Consommation : Le RTOS peut endormir le CPU quand aucune tâche n’est prête.
  • Stack : Chaque tâche a un stack dédié, à dimensionner et surveiller avec uxTaskGetStackHighWaterMark().

6. Bonnes Pratiques pour les RTOS en IoT

À FAIRE

  • • Dimensionner correctement les stacks de tâches.
  • • Utiliser des queues pour la communication inter-tâches.
  • • Garder les ISR très courtes.
  • • Protéger les ressources partagées avec des mutex.
  • • Ajouter des timeouts pour éviter les blocages.

À ÉVITER

  • • Appeler delay() dans les tâches au lieu de vTaskDelay().
  • • Accéder à des ressources partagées sans synchronisation.
  • • Créer une explosion de tâches (préférer regrouper les fonctions proches).
  • • Oublier de libérer mutex/sémaphores.
  • • Donner la même priorité à toutes les tâches.

Outils de Debug pour FreeRTOS

  • FreeRTOS+Trace : Visualisation des tâches en temps réel. [web:68]
  • uxTaskGetStackHighWaterMark() : Mesure du stack restant.
  • vTaskList() : Vue globale des tâches (si RAM suffisante).
  • GDB : Debug pas à pas avec support RTOS sur certains IDE. [web:65]
