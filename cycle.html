Le Cycle d'Instruction et le Temps Réel

Comment le CPU exécute une instruction et pourquoi la prévisibilité est vitale dans l'IoT.

Objectifs pédagogiques : Comprendre le cycle Fetch-Decode-Execute, analyser l'impact des sauts et interruptions sur les performances, et maîtriser les techniques d'optimisation pour garantir le déterminisme temporel dans les systèmes embarqués IoT.

1. Les Étapes du Cycle Fondamental (F-D-E)

Chaque instruction exécutée par le CPU suit un cycle bien défini, appelé cycle Fetch-Decode-Execute. Dans un système embarqué, la rapidité et la constance de ce cycle sont cruciales pour garantir les contraintes de temps réel.

Exemple concret : Mesure du temps d'exécution

Sur un STM32 (Cortex-M4 à 84 MHz), une instruction simple comme MOV R0, #5 prend 1 cycle d'horloge, soit environ 11.9 nanosecondes (1/84MHz). Pour une boucle de 1000 itérations, cela représente environ 12 microsecondes - un temps prévisible et mesurable. 

// Exemple : Mesure de cycles sur STM32
uint32_t start = DWT->CYCCNT;  // Compteur de cycles
// ... votre code ...
uint32_t end = DWT->CYCCNT;
uint32_t cycles = end - start;

1. Fetch (Recherche)

Le CPU récupère l'instruction à l'adresse indiquée par le Program Counter (PC) dans la mémoire (Flash ou RAM).

Détail technique : Sur ARM Cortex-M, le bus AHB (Advanced High-performance Bus) transfère 32 bits par cycle depuis la Flash. Si l'instruction fait 16 bits (Thumb), une seule lecture suffit.

2. Decode (Décodage)

L'unité de contrôle (CU) interprète l'instruction (ADD, MOV, JUMP, etc.) et détermine les données et les adresses nécessaires.

Détail technique : Le décodage identifie l'opcode (code opération), les registres source/destination, et les modes d'adressage. Sur RISC, cette étape est simple car les instructions ont un format fixe.

3. Execute (Exécution)

L'unité arithmétique et logique (ALU) effectue l'opération, écrit le résultat dans un registre ou en mémoire, et le PC est mis à jour.

Détail technique : Les opérations sur registres (ADD, SUB) prennent 1 cycle. Les accès mémoire (LDR, STR) peuvent prendre 2-3 cycles selon la latence de la RAM.

2. Impact des Sauts et Interruptions

Le flux d'instructions n'est pas toujours linéaire. Les sauts conditionnels (boucles, `if/else`) et, plus important encore, les interruptions peuvent casser le pipeline et introduire des latences. Dans l'IoT, cette latence doit être minimisée pour respecter le temps réel.

Les Sauts et le Pipeline

Lorsqu'un saut se produit, toutes les instructions qui avaient été pré-chargées dans le pipeline (étapes Fetch et Decode) deviennent invalides et doivent être purgées. C'est ce qu'on appelle une Stalle ou une Bulle de Pipeline, ce qui gaspille des cycles d'horloge.

Défi Temps Réel : L'incertitude du temps de stalle rend difficile de garantir une réponse dans un délai fixe (déterminisme).

Gestion des Interruptions (ISR)

Une interruption (ex: réception d'un paquet WiFi) suspend l'exécution du programme principal pour passer à l'Interrupt Service Routine (ISR). Le temps entre la demande d'interruption et le début de l'exécution de l'ISR est la latence d'interruption.

Exemple pratique :

Sur un ESP32, la latence d'interruption typique est de 2-5 microsecondes. Pour un capteur qui doit répondre dans les 10ms, c'est acceptable. Mais pour un contrôleur moteur nécessitant une réponse en 100µs, il faut optimiser l'ISR.

  • • Sauvegarde/Restauration : Sauver les registres et restaurer l'état du CPU après l'ISR ajoute des cycles.
  • • Vecteur d'Interruption : Le CPU saute directement à l'adresse de l'ISR (un type de saut critique).

3. Optimisation pour le Déterminisme

Pour assurer le déterminisme (réponse garantie dans un temps maximal), les développeurs IoT utilisent des techniques spécifiques :

Minimiser les ISR

Garder les routines d'interruption (ISR) extrêmement courtes pour réduire le temps de blocage.

Boucles Sans Saut

Utiliser des boucles simples ou des méthodes sans branchement complexes pour éviter les stalles de pipeline.

Compteurs de Cycles

Utiliser les registres de compteurs de cycles matériels (si disponibles) pour mesurer précisément la latence réelle.
